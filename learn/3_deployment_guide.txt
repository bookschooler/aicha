## 1. 배포와 백엔드의 쉬운 이해 (비유)
배포가 처음이신 분들을 위해 상점 운영에 비유해 보겠습니다.

- **프론트엔드 (데스크/진열대)**: 손님이 직접 보고 만지는 부분입니다. 주소를 입력하는 창과 그래프가 여기에 해당합니다. (Vercel에 배포)
- **백엔드 (주방/창고)**: 손님 눈엔 안 보이지만, 실제 요리를 하고 재료(데이터)를 꺼내오는 곳입니다. 주소를 받아서 상권 등수를 계산하는 로직과 CSV 데이터가 여기에 있습니다. (Render에 배포)
- **배포 (이전/개업)**: 지금은 이 모든 게 사용자님의 PC(집) 안에만 있습니다. '배포'는 이걸 **인터넷이라는 큰 시장(서버)으로 옮겨서 24시간 내내 다른 사람들도 볼 수 있게 개업**하는 과정입니다.

## 2. 플랫폼 연결 및 백엔드 배포란?
- **백엔드 배포**: 내 컴퓨터를 꺼도 프로그램이 계속 돌아가도록 24시간 켜져 있는 가상 컴퓨터(서버)에 백엔드 코드를 올려두는 것입니다.
- **플랫폼 연결 (GitHub 중심)**: 
    1. 사용자가 코드를 수정해서 **GitHub**에 올리면,
    2. 연결된 **Vercel/Render**가 "어? 코드가 바뀌었네?" 하고 알아차린 뒤,
    3. 자동으로 그 코드를 가져와서 새로운 버전으로 다시 설치해 줍니다. 
    → 즉, 사용자님은 GitHub에 코드만 올리면 배포 플랫폼이 알아서 서비스를 운영해 줍니다.

## 2. 배포 단계별 절차

### 단계 1: 소스 코드를 GitHub에 업로드
배포 플랫폼들은 GitHub와 연동되어 코드가 수정될 때마다 자동으로 업데이트를 진행합니다.
1. `github.com`에 새 저장소(Repository) 생성.
2. 현재 프로젝트 코드를 `push`.

### 단계 2: 백엔드(FastAPI) 배포 준비
백엔드 서버가 실행되기 위한 환경 파일들을 생성해야 합니다.
- **[MODIFY] api/requirements.txt**: `fastapi`, `uvicorn`, `pandas`, `pyproj`, `scipy` 등 필요 라이브러리 명시.
- **[NEW] api/Dockerfile**: (선택 사항) 일관된 실행 환경을 보장하기 위해 권장.

### 단계 3: 프론트엔드(React) 연동 주소 변경
현재 프론트엔드는 `localhost:8000`에서 데이터를 가져오도록 되어 있으나, 배포 후에는 백엔드가 설치된 실제 인터넷 주소로 변경해야 합니다.
- **[MODIFY] frontend/src/App.jsx**: API 호출 URL을 환경 변수(Vite Environment Variable)를 사용하여 관리하도록 수정.

### 단계 4: 플랫폼 연결
1. **Render.com** 접속 후 'Web Service' 생성 → GitHub 저장소 연결 → 백엔드 배포.
2. **Vercel.com** 접속 후 'Add New Project' → GitHub 저장소 연결 → 프론트엔드 배포.

## 3. 사용자 검토 및 주의사항

> [!CAUTION]
> **API 키 보안**: 주소 좌표 변환을 위해 외부 API(Kakao, Google 등)를 사용할 계획이라면, API 키가 코드에 직접 노출되지 않도록 '환경 변수(Environment Variables)'로 설정해야 합니다.

> [!IMPORTANT]
> **데이터 용량**: 현재 CSV 파일들의 용량이 크다면(예: 100MB 이상), 무료 티어 서버의 메모리 제한에 걸릴 수 있습니다. 이 경우 데이터를 SQLite 등 경량 데이터베이스로 변환하거나 데이터를 압축하는 최적화가 필요할 수 있습니다.

---
배포를 시작할 준비가 되셨다면, 첫 단계인 **배포용 환경 파일(`requirements.txt`) 생성**부터 진행하겠습니다.
